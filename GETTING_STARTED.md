# Getting Started

Welcome to Project Resonance.

This guide will help you understand the structure of the repository and how we work together as a product team.

Whether you are a contributor, reviewer, or simply exploring the project, this document should be your starting point.

---

# Project Overview

Project Resonance is an open-source iPhone companion for the NUX Mighty Plug Pro MP-3.

Our goal is to create an application that feels like an extension of the instrument rather than another piece of software.

Every design and engineering decision is guided by one principle:

> **Every decision should make it easier to create music.**

---

# Repository Structure

The repository is organized into several major areas:

| Folder | Purpose |
|---------|---------|
| PROJECT_BIBLE | Vision, philosophy, standards, and long-term direction |
| PRODUCT | Product documentation and feature specifications |
| DESIGN | Wireframes, mockups, UX reviews, and interaction design |
| ARCHITECTURE | Technical architecture and Architectural Decision Records |
| PRODUCT_DECISIONS | Product standards and decision logs |
| SPRINTS | Sprint planning and reviews |
| BACKLOG | Planned work and future ideas |
| PROJECT_HISTORY | Timeline, development log, and project history |
| TEMPLATES | Reusable documentation templates |
| assets | Project assets such as wireframes, diagrams, icons, and screenshots |

---

# How We Work

Project Resonance follows an iterative product development process.

Every significant idea should be evaluated before implementation.

Questions we ask include:

- Does it improve the guitarist's experience?
- Does it reduce friction?
- Does it align with our mission?
- Does it belong in Version 1.0?
- Should it be documented before implementation?

---

# Documentation Philosophy

Documentation is treated as a product.

Every document should:

- Reduce uncertainty.
- Be easy to understand.
- Have a clear purpose.
- Be kept up to date.
- Link naturally to related documents.

---

# Versioning

Project Resonance is developed incrementally.

Planning documents evolve through Version 0.x.

The Product Bible reaches Version 1.0 when Sprint 01 is complete.

The application itself progresses independently through prototype, beta, and release milestones.

---

# Git Commit Guidelines

Commit messages should use the imperative mood.

Examples:

- Add Product Bible cover
- Create repository index
- Document mission and vision
- Update glossary

Each commit should represent a meaningful unit of work.

---

# Development Workflow

Our typical workflow is:

1. Discuss an idea.
2. Evaluate whether it aligns with the product vision.
3. Decide where it belongs in the repository.
4. Document the decision.
5. Review and approve the documentation.
6. Commit the changes.
7. Implement when appropriate.

---

# Working Principles

Project Resonance values thoughtful progress over rapid progress.

We believe that investing time in understanding the problem leads to better solutions and a more maintainable product.

Whenever possible:

- Document before implementing.
- Prototype before polishing.
- Validate before expanding.
- Build for musicians first.

---

# Definition of Done

Work is considered complete when:

- Documentation has been reviewed.
- Decisions have been recorded.
- Related documents have been updated.
- Git commits have been made.
- The repository accurately reflects the current state of the project.
